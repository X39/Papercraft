# Documentation Roadmap

This roadmap tracks the move from the current README-based reference to a GitHub Pages user manual that can be handed to non-developer template authors.

The manual must be concept-led and sample-driven. Users need to understand what the template language concepts are, why they exist, and then how to use them. Most pages should combine:

- a short "What is this?" explanation,
- a short "When should I use this?" explanation,
- a short explanation,
- a tiny XML sample,
- the rendered output image when the result is visual,
- links to the deeper reference page.

Task pages should answer small user questions such as "How do I add a page number?", "How do I make every second table row gray?", or "How do I show a value from my data?". Concept pages should explain ideas such as areas, controls, template data, functions, transformers and styles before the reader sees too many examples.

The rendered images should be generated by tests, using the same approach as `test/X39.Solutions.PdfTemplate.Test/Samples/ReadMeSample.cs`. A documentation sample is only considered complete when the XML and output image can be regenerated from the repository.

## Definition of done

A manual page is complete when:

- It explains what the feature is and why a template author would use it.
- It answers real template-author tasks, not only API structure.
- Every visual feature has at least one generated PNG in the docs assets.
- Each sample XML is either generated by a test or copied from a test fixture.
- Attribute names and value formats are checked against source or tests.
- The page links to related beginner, reference and troubleshooting pages.
- The page renders correctly on GitHub Pages.

A sample is complete when:

- It demonstrates one concept only, unless it is in the complete examples section.
- It has a clear filename and test name, for example `Text_BoldAndCentered`.
- It writes deterministic output into a documentation asset folder.
- It avoids machine-specific output paths such as the current user's Desktop.
- It is small enough to understand without scrolling through a full invoice.

Implementation tasks:

- Add a `DocumentationSampleBase` derived from the existing sample approach.
- Render with `GenerateBitmapsAsync` and encode PNGs with SkiaSharp.
- Write images to `docs/assets/samples`.
- Use deterministic file names.
- Delete or overwrite stale files for the sample being rendered.
- Keep samples independent so one failing control does not block unrelated examples when debugging locally.

## Agent workflow

Agents should work in this order:

1. Pick one roadmap area or a small group of related "How do I..." samples.
2. Add or update documentation sample tests first when the topic has visual output.
3. Generate and check in the PNG assets.
4. Write the manual text around the generated result.
5. Update the roadmap status table and link the new page or sample.

Keep pull requests small. A good PR might add the text samples page, three sample tests and three generated images. A bad PR edits every chapter at once without generated outputs.

## Ground rules

- Write for template authors first. Avoid requiring C# knowledge unless the chapter is explicitly about integration.
- Prefer full XML snippets that can be copied into a template.
- Prefer many tiny examples over one large example.
- Include generated output images beside visual examples.
- Preserve technical accuracy by cross-checking README claims against tests and source code.
- Do not duplicate large blocks from the README without rewriting them into manual prose.
- Link between chapters instead of repeating reference tables everywhere.
- Use lowercase XML element names exactly as implemented.
- Mark uncertain behavior with a TODO and source reference rather than guessing.
- Every major chapter should start with "What this is" and "When to use it" before "How to use it".
