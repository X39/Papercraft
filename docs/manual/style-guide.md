# Documentation Style Guide

[Manual home](index.md) | [Documentation roadmap](work-plan.md)

Status: current.

## What Is This?

This guide keeps the manual consistent as new concept pages, task pages and samples are added.
It is written for documentation contributors and maintainers, not for template authors reading the manual.

The user-facing manual should stay concept-led and sample-driven.
Each page should help a non-developer template author understand what a template feature is,
when it is useful and how to try the smallest safe example.

## When Should I Use This?

Use this guide before adding or expanding manual pages.
It applies to concept chapters, task recipes, control reference pages, complete examples and troubleshooting notes.

If this guide conflicts with verified behavior in source code or tests, keep the verified behavior and update this guide.
If behavior is unclear, add a TODO that names the source or test area that needs verification.

## How Do I Start?

Start every major concept page with these sections:

- `What Is This?`
- `When Should I Use This?`
- `How Do I Start?`

Keep the first example tiny.
Prefer several short XML snippets over one large template unless the page is in the complete examples chapter.

## Audience And Tone

Write for template authors who edit XML but do not need to understand .NET integration.
Avoid C# service registration, dependency injection and extension-point details outside the developer appendix.

Use direct, task-oriented language.
Prefer "Use `footer` for repeated content at the bottom of every page" over implementation descriptions.
Explain terms before using them heavily.

## Examples

Examples should demonstrate one idea at a time.
Use complete XML templates when a reader is expected to copy the whole sample.
Use small fragments only when the surrounding page already explains where the fragment belongs.

Every XML sample must be one of these:

- generated by a documentation sample test,
- copied from an existing test fixture or README section that is named near the sample,
- marked with a TODO that names the source or test area that needs verification.

Prefer lowercase XML element names exactly as implemented.
Do not invent attributes, values, functions or transformer syntax.

## Visual Samples

Visual examples need generated preview assets under `docs/assets/samples`.
Add or extend documentation sample tests before adding images.
Generated samples should use deterministic file names and avoid machine-specific paths.

When a page includes an image, name the sample test near the XML so future contributors can regenerate it.

## Page Shape

Use short pages with links to deeper references instead of repeating long tables everywhere.
Task pages should answer one user question, such as how to draw a line or show a value from data.
Reference pages may contain attribute tables, but examples should still be small.

Every user-facing page should include navigation back to the manual home and adjacent chapters or related pages.
Use relative links so pages render correctly on GitHub Pages.

## Controls Reference Structure

Keep [Controls](controls.md) as the controls landing page and shared orientation point.
Split detailed built-in control references into separate pages when each control is documented.

Use this pattern for control pages:

- one page for text,
- one page for border and shared box styling,
- one page for lines,
- one page for images,
- one page for page numbers,
- one page for tables and table children,
- one page for charts and chart data,
- one page for rich text, links and inline text,
- one page for flow helpers such as blocks, spacers, page breaks and columns,
- one page for lists and list items,
- one page for form-like or signing helpers such as checkboxes and signatures,
- separate optional-control pages for QR and ZXing barcode packages.

Keep shared layout attributes in [Layout fundamentals](layout-fundamentals.md) and link to them from control pages.

## TODOs

Use TODOs for uncertainty, not as a substitute for verification.
A useful uncertainty note names the missing verification target, for example:

`Verify accepted table column width syntax against source/X39.Solutions.Papercraft.Core/Controls and test/X39.Solutions.PdfTemplate.Test/Controls.`
